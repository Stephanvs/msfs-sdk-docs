Introduction

As one might expect, Ground Vehicles are SimObjects representing vehicles that move on the ground. Those SimObjects can be used in missions, for example, but are mainly used as part of the Living World, to populate airports. More precisely, there are two types of Ground Vehicles uses in airports: the Ambient Traffic, which are vehicles random wandering accross the airport while following taxi paths, and the services, which can directly interact with aircrafts for specific purposes. The latter generally require additionnal specific configuration in order to work properly, and have thus their own chapters here. Nevertheless, the following sections detail the basics of Ground Vehicle configuration for all types of Ground Vehicles, may they be part of Ambient Traffic or Services vehicles.

Chapters specific to Services vehicles are available here: BaggageCart, BaggageLoader, BoardingRamp, CateringTruck, FuelTruck, GroundPowerUnit, Pushback.

Basics of configuration

As any other SimObject, a Ground Vehicle main folder generally contain a model folder, a texture folder, and a sound folder, or possibly several of these (with a “.something” suffix) for variants. Moreover, the model folder contain an xml file referencing LoDs and ModelBehaviors, as for any SimObject, and as described in the corresponding chapter Contrary to aircrafts however, Ground Vehicles have a single .cfg file at their root folder, which is sim.cfg (while aircrafts have a main aircraft.cfg complemented by a number of other cfg files).

The sim.cfg for a Ground Vehicle should always contain the following sections and parameters:

[VERSION]
      Major=1
      Minor=0

The version number of 1.0 is to ensure vehicles are up-to-date with the base version of MSFS. Without any specified version number, 0.0 is assumed, and the vehicle should keep some functionnality, for FSX backward-compatibility, but with lack some new improvements of MSFS, such as ModelBehaviors.

[fltsim.0]
      title=Choose_the_UNIQUE_title_of_your_choice_here

Titles are unique identifier of SimObjects and their variants. We recommend using your company name as a prefix, to guarantee the uniqueness of your SimObject even when included with objects produced by other 3rd party developpers. This section can also contain “model=”, “texture=”, and “sound=” to specify folders, and other sections [fltsim.1] and so on for variants.

[General]
      category=GroundVehicle

This is mandatory to specify this SimObject is, in fact, a Ground Vehicle.

[DesignSpecs]
      max_speed_mph = somePositiveValue

In order for the GroundVehicle to be able to move, it needs to have a strictly positive maximum speed defined in this section.

Other sections of sim.cfg

[fltsim.*]

The first such section must be [fltsim.0], and optional other such sections must have increasing numbers without skips (you cannot have a [fltsim.2] without a [fltsim.1] first, for example). Every such section must contain a parameter title with a different unique title for each section (corresponding to each variant).

Then, every such section can contain parameters model, texture, and/or sound, in order to change the folder used for this variant. By default, each parameter as a value of "" (empty string). The folder used will be those with the exact name model, texture, sound in case of empty string, otherwise they will be model.*, texture.*, sound.* if the parameters are some non-empty string. The following example will use folders model.something, texture.42, and sound

[fltsim.1]
      title=Title_Of_First_Variant
      model="something"
      texture="42"
      sound=""

[General]

The General section must be present, and contain the parameter category=GroundVehicle.

It can also contain parameter DistanceToNotAnimate to define a distance, in meters, that defaults to 1000m. When the vehicle is farther away from the viewpoint of the camera of MSFS, its ModelBehaviors and other animations are not executed, to improve general performances. If the camera gets farther than this distance during an animation, then comes back closer, the animation will immediately skip to the frame it should have been without the interruption, and for ModelBehaviors, the simvar ANIMATION DELTA TIME will contain the total time of interruption instead of the current frame delta time. This is means that, as long as the animations/ModelBehaviors are not visible from this distance, this performances optimization will have no visible impact whatsoever.

[PILOT]

The optional section determines how to place a Human Character as a pilot (or driver) of this vehicle.

Parameter pilot_attach_node contains a string which is the name of a Node in the model of the vehicle, where the pilot will be attached. the default value is “PILOT_0”, though we recommend defining it explicitely in sim.cfg anyway.

Parameter pilot_default_animation contains a string representing the animation (BlendTreeState) that will be used by the Human Character pilot. This is especially useful if the Human Character model used is standing by default, in order to make sure he is properly sit down in the vehicle. The models of Human Characters used in the vehicles are determined by the GroundVehiclePilots section of the Living World configuration file. The name of the “animation” specified here must be the name of a proper BlendTreeState for the model of Human Character used as pilot ; more details in the AnimGraph chapter. If BlendTree animation layers are used by the Human Character, the pilot_default_animation can actually contain more data, in the form pilot_default_animation = A,B,C,D,E,F,..., where A is the name of a BlendTreeState, B the layer containing this state (default to “Base”), C is another BlendTreeState with D its layer, and so on… Each of these BlendTreeStates must belong to different layers.

[contact_points]

Although the Ground Vehicle could work fine with the default values, we recommend including this section explicitely anyway. It contains two parameters static_cg_height and static_pitch, both default to 0.

Parameter static_cg_height contains the distance, in feet, between the center of the vehicle model and the ground, positive if the center of the model is above the ground. Note that with a version number less than 1.0 (or unspecified), static_cg_height should actually contain this distance + 1, due to FSX backward-compatibility, even though this is rather counter-intuitive.

Parameter static_pitch contains the pitch of the vehicle when at rest on flat ground, in degrees.

[DesignSpecs]

This section contain a parameter max_speed_mph (that can be replaced by parameter max_speed_knots) that is necessary for the vehicle to be able to move, and two optional parameters acceleration_constants and deceleration_constants (that have default values).

The parameter max_speed_mph or max_speed_knots express the maximum speed that can be reached by the vehicle while moving forward on flat ground (it can actually go faster when driving downhill or falling off a cliff). Guessing which units are used in the example “max_speed_mph = 115” is left as an exercise for the reader.

Both acceleration_constants and deceleration_constants have two comma-separated values, which default to 1.0, 0.5. As their names suggest, acceleration_constants are used to control the acceleration of the vehicle, and deceleration_constants its deceleration. The first value is a time constant of effect responsiveness (higher value for faster reactions), while the second one is a maximum acceleration in Gs.

[WAYPOINT]

This section contains various parameters controlling the AI that drives the Ground Vehicle along waypoints, all of which are optional due to default values.

Parameter SlowDownDistance is a distance in meters, at which the vehicle start slowing down before a sharp turn or its destination. Default value is 50m.

Parameter StopDistance is a distance in meters from the destination, at which the vehicle consider it has reached its destination, and start actually braking to reach a full stop.

Parameter SlowDownOnLastWayPoint is a boolean (1 for true, 0 for false), determining if the vehicle should start really slowing down (but not trying to fully stop either), on the last 5 meters before its destination. Default to true.

Parameter FrontRadiusMeters is a length in meters from the center of the vehicle to the front of it. It is used by the AI to better manage its turns. By default, or if set to 0, an approximate exagerated value is computed by using a bounding box on the model.

[LIGHTS]

This optional section contain a list of newlight.* starting from newlight.0, with increasing indexes (without skips). Each line describes some kind of light on the vehicle, whether an emissive texture or a FlightSimLight in the model (or both), in order to be able to turn them on or off.

A parameter of the form newlight.0 = A,B,C,D has 4 string values, where A is actually from an enum describing the type of light, B is the name of a FlightSimLight Node, C the name of a node containing an emissive material whose name is in D. The type of light (first value of the line) must be one of the following : “HeadLight”, “BackLight”, or “GyroLight”. Here is a short example :

newlight.0 = HeadLight, MyLightName, SomeNodeName, MaterialNameInThisNode
      newlight.1 = HeadLight, MySecondLight, , 
      newlight.2 = BackLight, , SomeNodeNameOrAnother, DifferentMaterialWithEmissive

As shown in this example, it is possible to have multiple lights of the same type, and to have lines without the second value (the FlightSimLight) or without the third and fourth values (node+material with emissive).

ModelBehavior helpers

Several templates useful for Ground Vehicles are available in the package fs-base-aircrafts-common, one of the core packages of MSFS. They can be accessed by including the following line in the ModelBehaviors section of the xml of a vehicle:

<Include Path="Asobo\Misc\GroundVehicles.xml"/>

Here is an example of use of these helpers:

<ModelBehaviors>
          <Include Path="Asobo\Misc\GroundVehicles.xml"/>
          
          <Component ID="Wheels">
              <UseTemplate Name="ASOBO_Wheel_Roll">
                  <ANIM_NAME>Wheel_Rotation</ANIM_NAME>
                  <WHEEL_RADIUS>0.364</WHEEL_RADIUS> <!-- in meters -->
              </UseTemplate>
              <UseTemplate Name="ASOBO_Wheel_Turn">
                  <ANIM_NAME>Wheel_Orientation</ANIM_NAME>
                  <ORDER_MULTIPLIER>-1</ORDER_MULTIPLIER> <!-- 1 if first frame of animation is turning left, -1 if turning right -->
                  <PERCENT_PER_SECOND>50</PERCENT_PER_SECOND> <!-- speed of the anim in percent per second. 100 means from max left to max right in 1 second -->
              </UseTemplate>
          </Component>
          
          <Component ID="Lights">
              <UseTemplate Name="ASOBO_GroundVehicle_HeadLights"/>
              <UseTemplate Name="ASOBO_GroundVehicle_BackLights"/>
              <UseTemplate Name="ASOBO_GroundVehicle_Gyrolights_Animated">
                  <ANIM_NAME>Gyrolight</ANIM_NAME> <!-- Anim for rotating the light -->
                  <PERCENT_PER_SECOND>100</PERCENT_PER_SECOND> <!-- which percentage of the anim is played in 1 second. 200 means anim is played twice per second -->
              </UseTemplate>
          </Component>
      </ModelBehaviors>

ASOBO_Wheel_Roll

The template ASOBO_Wheel_Roll helps to play an animation of rolling wheels for the vehicle, adapting the speed of the animation to the speed of the vehicle at all time. It takes two mandatory parameters: ANIM_NAME, the name of the animation that “rolls” the wheels, and WHEEL_RADIUS, the radius in meters of those wheels.

The animation must be linear, and be exactly one full rotation (360°) of the wheel (first frame and last frame of animation being different, and not a “duplicate”).

For vehicles having several wheels of different size, we recommend creating an animation for each size of wheels, and using as many ASOBO_Wheel_Roll templates, each with the correct radius.

ASOBO_Wheel_Turn

The template ASOBO_Wheel_Turn helps to play an animation of wheels turning left or right when the vehicle itself turn left or right. It takes three mandatory parameters: ANIM_NAME, the name of the animation, ORDER_MULTIPLIER, specifying if animation has been created “left to right” or “right to left”,and PERCENT_PER_SECOND, the maximum speed of the animation in case the vehicle turns suddenly from sharp left to sharp right (or the other way around).

The animation should start at the angle for a sharp turn left, rotate until being straight in the middle frame, and keep rotating until sharp right angle for the last frame. The ORDER_MULTIPLIER should then be 1 ; the animation can also be from right to left instead, in which case the ORDER_MULTIPLIER should be -1.

The PERCENT_PER_SECOND parameter can be higher than 100 if desired, but be warned that the Ground Vehicle AIs can sometimes switch from being turning right with a sharp angle to left with a sharp angle in a single frame, which means the animation would visibly “jump” from one side to the other if the PERCENT_PER_SECOND is too high, leading to almost no “smoothing”.

ASOBO_GroundVehicle_HeadLights

The template ASOBO_GroundVehicle_HeadLights controls the HeadLights of the vehicle, turning them on or off when appropriate. It has no parameter.

The vehicle needs to have its Headlights defined in sim.cfg, as explained here. The headlights will then be automatically turned off when the vehicle turns off its engine, and during the day.

ASOBO_GroundVehicle_BackLights

This behaves exactly as ASOBO_GroundVehicle_HeadLights, but for BackLights instead of HeadLights.

ASOBO_GroundVehicle_GyroLights

This behaves similarily to ASOBO_GroundVehicle_HeadLights, but for GyroLights instead of HeadLights. Although, GyroLights stay on during the day (while the engine is on), contrary to HeadLights and BackLights.

ASOBO_GroundVehicle_GyroLights_Animated

This behaves similarily to ASOBO_GroundVehicle_GyroLights, with some addition. It requires two parameters: ANIM_NAME, the name of an animation to play while Gyrolight is on, and PERCENT_PER_SECOND, determining the speed of the animation.

This can be useful for example to have a rotating animation for the light. The animation should be able to loop smoothly, and will be played at all time when the Gyrolight is on, which is when the engine of the vehicle is on.

A value of PERCENT_PER_SECOND of 100 means the animation will be played in 1 second before looping back to the start. 50 means the animation will take 2 seconds, while 200 would mean the animation is played in 0.5 second.
