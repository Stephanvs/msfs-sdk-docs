<SimBase.Document>

Base of an xml document.

Element Description Unique Required
<Checklist.Checklist> Yes No
<Checklist.CheckpointLibrary> Yes No
Attribute Description Type Required Default
Type Type of the xml file, either “Checklist” or “CheckpointLibrary”. string Yes

Note: This element must contain an unique child element corresponding to value of the Type attribute.

<Checklist.Checklist>

Base of an aircraft checklist file, as only element of a SimBase.Document. Can include Libraries, and must contain one or several steps (that will each contain pages of checkpoints).

Element Description Unique Required
<IncludeCheckpointLibrary> A checkpoint library to be loaded.
Checkpoints from said library can then be used by using a ReferenceId		 No No
<Step> A list of pages.
Used to regroup one or several pages by logical steps.		 No Yes

<Checklist.CheckpointLibrary>

Base of a CheckpointLibrary file, as only element of a SimBase.Document. Contains only a list of checkpoints.

Element Description Unique Required
<Checkpoint> A single item of Checklist, with an Id to be referenced in aicraft checklist files. No Yes

<IncludeCheckpointLibrary>

Attribute Description Type Required Default
FileName Path of an xml file for a checkpoint library. string Yes

<Step>

Attribute Description Type Required Default
ChecklistStepId A key from Checklist Steps list, defined by missions propsets.
Used to reference the step by a mission.		 string Yes
Element Description Unique Required
<Page> A list of checkpoints, displayed in the UI with a title.
Used to regroup one or several checkpoints in a single page of this checklist.		 No Yes

<Page>

Attribute Description Type Required Default
SubjectTT A translation key to the page title. See Localization for more details. string Yes
Element Description Unique Required
<Checkpoint> A single item of a checklist. No No
<Block> A block of checkpoints, with a title, and possibly a condition. No No

Note: A page must contain at least one checkpoint, or a block containing at least one checkpoint. Order between checkpoints that are direct elements of the page, and blocks of checkpoints, is important.

<Block>

Attribute Description Type Required Default
SubjectTT A translation key to the block title. See Localization for more details. string Yes
Element Description Unique Required
<BlockCondition> A condition to execute the block’s checkpoints. In non-Manual mode, these checkpoints will be skipped if condition is false when their evaluation would start. Yes No
<Checkpoint> A single item of a checklist. No No
<Block> A block of checkpoints, with a title, and possibly a condition. No No

Note: Order between each element, checkpoint or block, is important. A block without blockCondition will always be evaluated.

<BlockCondition>

Element Description Unique Required
<Val> Value that can be defined various ways (see corresponding section). The value will be read as a boolean, meaning 0 is false and non-zero is true. Yes No
<Operator> Operation that will be evaluated using one or several values (see corresponding section). Yes No

Note: A blockCondition must contain either a val or an operator.

<Checkpoint>

Attribute Description Type Required Default
ReferenceId Id of a checkpoint from a loaded Library.
Used to load a shared checkpoint for an aircraft checklist.		 string No
Id Id of a checkpoint in a Library.
Set in Library files, and used as a reference in aircraft checklist files.		 string No
Element Description Unique Required
<CheckpointDesc> Text description of the checkpoint, that is displayed on in-game UI. Yes Yes
<Sequence> Sequence of tests required to validate the checkpoint. Yes No
<Test> Single test required to validate the checkpoint Yes No
<Action> Actions to be executed when the checkpoint start being evaluated, or at the end of its validation. No No
<SetTemplateValue> Defines a value that can then be used inside the checkpoint’s other elements.
Mostly used to add parameters to a library checkpoint that can be overridden in a checklist checkpoint.		 No No
<SetTemplateAction> Defines an action that can then be used inside the checkpoint’s other elements.
Mostly used to add parameters to a library checkpoint that can be overridden in a checklist checkpoint.		 No No
<Instrument> Reference to an instrument associated with this checkpoint.
This instrument can be highlighted in-game.		 No No
<ForceCamera> Specify a camera to be used when highliting instruments, instead of using the regular instruments camera. Mostly useful when multiple instruments that are far apart need to be highlighted. Good practice is to put ForceCamera in a checklist file, not in a library. Yes No
<Feedback> Text to be displayed in the interface while evaluating the checkpoint, in non-Manual mode. Yes No
<Clue> Text to be displayed in the interface while checkpoint is selected. Yes No

Note: Only one of sequence and test should be defined. If both are defined, one of them will be ignored. If neither is defined, the checkpoint will be automatically validated in all modes except Manual.

Note: All elements except checkpointDesc and clue are ignored while in Manual mode.

<CheckpointDesc>

Attribute Description Type Required Default
SubjectTT A translation key to the checkpoint subject. See Localization for more details. string Yes
ExpectationTT A translation key to the checkpoint expected action or state. See Localization for more details. string Yes

<Feedback>

Attribute Description Type Required Default
Name A translation key to the feedback text. See Localization for more details. string Yes
DurationTimer If true, will display the remaining total Duration time in real-time, as seconds, in the text in place of a %f. boolean No
SimVarName If defined, will display the value of the corresponding simvar, in the text in place of a %f. string string Yes
Units Units of the simVar. Must be defined if SimVarName is defined. string Yes

Note: This text will be displayed when the checkpoint being evaluated, in Automatic or Copilot mode.

Note: A Feedback element can contain at most one of the following (sets of) attributes: - DurationTimer - SimVarName and Units

When either is defined, if the translated text contain a %f, it will be replaced by the corresponding value (but only the first %f). You can also use %.0f to display 0 digits after the decimal, and similarly %.1f for 1 digit after the decimal, %.2f…

<Clue>

Attribute Description Type Required Default
Name A translation key to the clue text. See Localization for more details. string Yes

Note: This text will be displayed when the checkpoint is clicked in Manual mode, or in non-started Automatic mode. In started Automatic mode, it will also display if currently for evaluated checkpoint, if it does not have a Feedback text.

<Sequence>

Attribute Description Type Required Default
SeqType Type of sequence to be used.
Possible values are: Ordered, Unordered, Parallel.		 enum Yes Unordered
Element Description Unique Required
<Sequence> A sub-sequence of tests. No No
<Test> Contains the logic for a single test to perform. No No
<Once> Determines whether this sequence only needs to be validated once to then be considered forever validated. Yes No
<Duration> Duration during which all the elements of the sequence need to be valid for it to be validated. Yes No
<Instrument> Reference to an instrument associated with this sequence.
This instrument can be highlighted in-game.		 No No
<Action> Actions to be executed during evaluation of this sequence, depending on various conditions. No No

SeqType

Regardless of the type of sequence, all tests or sub-sequences in this sequence need to be validated simultaneously (or have a once True and an earlier validation) for the main sequence to be valid. Informally, an Ordered SeqType means the sequence is supposed to be executed in strict order (determined by the order of elements), an Unordered SeqType means the sequence can be executed in any order, and a Parallel SeqType means the sequence elements should be executed simulatenously. More details:

SeqType Description
Ordered Every test and sequence element of this sequence is ordered as in the xml. Whenever the sequence is evaluated, each of those element is evaluated in order, from the first one up until an element that is evaluated to False. Any later element is thus not evaluated during this frame, and their actions not executed. Next frame evaluation will start over from the first element.
Parallel Whenever the sequence is evaluated, every test and sequence element of this sequence is evaluated (in order of their xml order) during the same frame.
Unordered Without Copilot mode, the evaluation of such sequence works exactly as the Parallel one: every element is evaluated. However, in Copilot mode, the evaluation will follow the Ordered SeqType. This difference is important to avoid a copilot doing everything (especially actions) at the same time, while still permitting User without Copilot mode to complete this sequence in any order.

Note: Frame, as used in the descriptions above, refer to a portion of time during which some computation are done by the game, such as evaluation of various tests and sequences. Usually, one second is composed of at least 30 consecutive frames. Actions and test evaluation can be executed at most once per frame.

Element Description Unique Required
<Val> Value that can be defined various ways (see corresponding section). The value will be read as a boolean, meaning 0 is false and non-zero is true. Yes No
<Operator> Operation that will be evaluated using one or several values (see corresponding section). Yes No

<Val>

Attribute Description Type Required Default
Value Constant value. number No
SimVarName Simulation Variable name, as defined here. Must be defined with units. string No
Units Units name, as defined here. Must be defined with simVarName. string No
Code Allows for more complex evaluation, using the ESP XML gauges code syntax. string No
ReadVar Read the value previously stored in the checklist-specific variable with this name. string No
ATCState Name of an ATC State.
Used for comparison with an ATCParam State.		 string No
ATCParam Get the current ATC State or the last ATC message ID.
Used for comparison with an ATCState Val, or a constant integer value.		 enum: State or LastMessageId No

<Operator>

Attribute Description Type Required Default
OpType Type of operator, as detailed in the table below. enum Yes Equal
Tolerance Numerical tolerance only used for the Equal OpType.
The comparison will then be “Is the difference between the values less than or equal to tolerance?”.		 number No 0
Element Description Unique Required
<Val> Unit value. No No
<Operator> Sub-operation, evaluated into a value. No No

Note: An Operator must contain at least a val or an operator.

Note: The relative order between val and operator elements is important, and they are all considered “values” similarly. The required number of “values” elements depends on the OpType, as follows:

OpType Description Number of values
EQUAL Returns true (1) if both values are equal, false (0) otherwise.
If a tolerance is defined, returns true if abs(v1-v2) <= tolerance, instead.		 2
LESS Returns true (1) if first value is (strictly) less than second value, false (0) otherwise. 2
GREATER Returns true (1) if first value is (strictly) greater than second value, false (0) otherwise. 2
LESS_OR_EQUAL Returns true (1) if first value is less than or equal to second value, false (0) otherwise. 2
GREATER_OR_EQUAL Returns true (1) if first value is greater than or equal to second value, false (0) otherwise. 2
BETWEEN Returns true (1) if first value is greater than or equal to second value and less than or equal to third value, false (0) otherwise. 3
NOT Returns true (1) is value is false (0), and returns false (0) otherwise. 1
AND Returns true (1) if both values are true (non-zero), false (0) otherwise. 2
OR Returns true (1) if at least one of the two values is true (non-zero), false (0) otherwise. 2
PLUS Returns the sum of the two values. 2
MINUS Returns the first value minus the second value. 2
MULTIPLY Returns the product of the two values. 2
DIVIDE Returns the first value divided by the second value. 2

<Once>

A <Once> element must have a single boolean value as follows:

<Once>True</Once>
      <!-- OR -->
      <Once>False</Once>

If, during its evaluation, a test or sequence with a once True is validated (including its possible duration), then it will remain validated until Reset, and will never be evaluated again. This means actions within this test or sequence will not be executed in later either.

<Duration>

If a test or sequence is validated but has a non-zero Duration, it will not be considered validated until the Duration criteria has also been validated.

Attribute Description Type Required Default
Value Number of seconds. number Yes
Cumulative Determines if the duration is accumulated on times where the test is Validated even if interrupted by some non-valid time, or if the Duration timer is reset is the test becomes non-validated. boolean Yes

<Instrument>

Attribute Description Type Required Default
Id Id of an instrument, defined as PartID in ModelBehaviours of corresponding aircraft. string Yes

Instruments for an aircraft are defined in an xml file describing the interior of the 3D model. In this file, ModelBehaviors are defined and use Components associated with parts of the model, that can have a PartID defined. This PartID can then be used as a reference in a Checkpoint to highlight said instrument. A Component with a PartID should also have a CameraTitle and a Node to be able to properly highlight the instrument, and to change the camera to be able to see the instrument more easily.

See section about Model Behaviors for more details.

<ForceCamera>

Attribute Description Type Required Default
Name Name of the camera to be used in this checkpoint, regardless of instruments camera, when highlighting instruments. string Yes

Note: The name of the camera correspond to the “Title” as defined in the CameraDefinition parts of aircraft.cfg file. Good practice is to use ForceCamera only in checklist files, not in checkpoint libraries.

Condition

Enum values Description Where can it be used?
IfTestValueTrue Only if the evaluation of testValue is currently True. If Used in a Sequence, uses the value of all sub-tests as a TestValue. Only in actions element of <Test> or <Sequence>
IfTestValueFalse Only if the evaluation of testValue is currently False. If Used in a Sequence, uses the value of all sub-tests as a TestValue. Only in actions element of <Test> or <Sequence>
Init Only when the checkpoint start being evaluated for the first time since Reset. Only in actions element of <Checkpoint>
End Only when the checkpoint is completely validated. Only in actions element of <Checkpoint>
Any If defined in an action that is an element of <Test> or <Sequence>, executed every time the testValue is evaluated.
If defined in an action that is an element of checkpoint, executed at Init and End of said checkpoint		 Every action

SpecialAction

Enum values Description Where can it be used?
ForceValidate Force the Validation of this test, ignoring possible durations or testValue not valid.
Mainly used for Copilot mode, to ignore an unimportant test that cannot be completed by copilot.		 Only in actions element of <Test>
WaitForUserToDoIt Waits until the test is validated.
Mainly used for Copilot mode, to force the copilot to wait even if no Copilot action is defined (otherwise the default Copilot behaviour is to force validate this test).		 Only in actions element of <Test>
WaitUntilTestValueValidated Waits until the test is validated.
Mainly used for Copilot mode, to force the copilot to wait even if no Copilot action is defined (otherwise the default Copilot behaviour is to force validate this test).		 Only in actions element of <Test>
WaitForDuration Once this action triggers (depending on condition), current test cannot be validated (regardless of duration or testValue) until some time has passed.
Time (in seconds) must be specified in the value attribute of this action.		 Only in actions element of <Test>

Note: WaitForUserToDoIt and WaitUntilTestValueValidated work the same way (currently), but it is recommended to use the right one for clarity, because they have different meaning. WaitForUserToDoIt means the User should manually do some action, and the copilot will not (or cannot) do it, and will not skip it either. WaitUntilTestValueValidated means the only required action is waiting; user and copilot do not have any action to perform, and this should not be skipped either.

StoreVar and ReadVar

StoreVar and ReadVar can be used to create variables that are specific to the Checklist and use them. Those variables are destroyed whenever a Checklist page is Reset (either by pressing the “Reset Page” button, or by restarting the mission). Names for those variables can be any string. Values stored and read are numbers. Booleans (true/false) are represented by 0 (false) or 1 (true). Note, however, that any non-zero number will evaluate to true when used as a boolean.

ReadVar can be used in a <Val> to retrieve a previously stored variable. If the requested variable has never been stored, it will return 0 instead.

StoreVar can be used in an <Action> to store a value in a variable. The corresponding action must contain one of the following: - a <Val> element - a <Operator> element - a Value attribute, containing a constant number

If multiple are defined, behavior is undefined. If none are defined, 0 will be stored by default (erasing any previous value stored in this variable).
