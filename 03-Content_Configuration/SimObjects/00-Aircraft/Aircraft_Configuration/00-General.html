Introduction

Every Microsoft Flight Simulator aircraft is defined by a mandatory Aircraft Configuration File, or aircraft.cfg. It mainly provides general information about the aircraft and its variations.

For now, please look at the ESP documentation‘s Aircraft Configuration File for missing sections’ parameters. Most (if not all) of the parameters described there are valid for Microsoft Flight Simulator.

Sections

[VERSION]

All configuration files without a version specified are versioned as 0.0 (major.minor). This includes legacy aircraft.cfg files imported from previous versions of the game.

Setting this version number to 1 or higher does a number of things: - It allows splitting of the aircraft configuration file sections. This is optional, meaning all sections described in the other configuration file chapters may be defined in the aircraft.cfg. - It completely removes the .air file as a dependency. All parameters must be configured in the configuration files instead. Note that this is also possible with lower versions, though in this case the .air file will be read for missing parameters. - It enables a number of improvements on animation and/or simulation. An incomplete list: - Model animations that are configured with lag (see the Model Behaviors section) will ignore this lag variable on their first frame of animation, immediately initializing to the requested state. - Turbine engines can now generate some torque from their starter rather than solely rely on combustion. - Turbine engines starters can be automatically disabled after reaching a specified N1 or N2 speed. - Flaps can now have a position in between their defined positions. - Electrical fuel pumps may be set to auto in addition to on or off.

Property Description Type Unit Range Mandatory
major Major CFG file version number Integer Number >= 0 No
minor Minor CFG file version number Integer Number >= 0 No

[GENERAL]

Configuring an aircraft for air traffic and multiplayer

In the general section of the Aircraft Configuration File there are multiple entries that relate to properly configuring a plane to be integrated in multiplayer, or as a grounded/parked plane, or part of the air traffic. They are mostly related to how the plane is identified by the database of the International Civil Aviation Organization, or ICAO for short, and they all must be non-localised.

Property Description Possible Values Examples
icao_type_designator The ICAO type designator of the plane. It could be different from the atc_type field. Only possible values are on the ICAO official database. Airbus A320 Neo (icao_type_designator = “A20N”)
icao_manufacturer The ICAO manufacturer of the plane. It could be different from the actual manufacturer of the plane. Only possible values are on the ICAO official database. Textron 172 Skyhawk (icao_manufacturer = “CESSNA”)
icao_model The ICAO model name of the plane. It could be different from the atc_model field. Any well-formed model name is accepted. Textron 172 Skyhawk (icao_model = “172 Skyhawk”)
icao_engine_type The ICAO engine type of the plane. This could be different from the atc_model field. Piston, Turboprop/Turboshaft, Jet, Electric, Rocket Daher TBM 930 (icao_engine_type = “Turboprop/Turboshaft”)
icao_engine_count The ICAO engine count of the plane. Any integer, standard is between 1 and 8. Daher TBM 930 (icao_engine_count = 1)
icao_WTC The ICAO Wake Turbulence Category of the plane. L, L/M, M, H Daher TBM 930 (icao_WTC = “L”)
icao_generic If true (1), the plane is intended to be only used as a passive/Air Traffic plane and is not identifiable as a specific real plane. 1, 0(Default) Generic Airliner Twin Engines (icao_generic = 1)

For multiplayer, the user might be choosing to use optimized and generic planes to represent other players’ planes, or they might not possess the correct plane model. In those cases, if the icao_type_designator matches an entry in the ICAO database, the model will be swapped with a generic plane that is as close as possible to that plane, in terms of WTC, engine count and engine type.

For air traffic, if the icao_type_designator matches an existing entry in the ICAO database, it might be used to represent a plane from the Real Online traffic. Please check the FLTSIM section and the AI configuration file documentation for further information on correctly integrating the plane in the grounded/parked planes and in air traffic.

[SERVICES]

This section details which Airport Services can be available for this aircraft. Each line in the section contain a name of a service, followed by = 1 to mark it as avaible, or = 0 to make it unsuitable for the aircraft. Any unspecified service is considered unavailable by default. The following list of services exist:

Service name Additional details
FUELTRUCK Ability to call a FuelTruck from a parking. Even if unavailable, this does not prevent the aircraft to taxi to a Fuel Station
BAGGAGE_LOADER The whole baggage service is determined by this
CATERING_TRUCK
BOARDING_RAMP i.e. movable stairs
GROUND_POWER_UNIT
PUSHBACK Large model of pushback, mostly for airliners.
SMALL_PUSHBACK Small model of pushback vehicle, designed for small GA aircrafts and private jets. The aircraft needs to have a centered front wheel
MARSHALLER
JETWAY If unavailable, aricraft will still be able to park at Gates, but not to connect to a jetway

Note that most services require properly defined [Interactive Points][interactivepoints] in order to be useable.

Also note that if neither PUSHBACK nor SMALL_PUSHBACK are available (for example with aircrafts having a left and right front wheels instead of a centered one), the User will still be able to pushback the aircraft (with Shift+P), but this might appear as “magic” because no visible vehicle or human character will be seen pushing the aircraft.

[EFFECTS]

The effects section of the file refers to the visual effects that result from various systems or reactions of the aircraft. An effect file associated with a keyword in this section will be used when the corresponding action is triggered. If no entry is made a default effect file will be used. The table below outlines the aircraft effects currently supported, though of course not all effects are supported on all aircraft.

Each entry can be followed by a 1 if the effect is to be run for a single iteration. Set this number to zero or leave blank (the default), for the effect to continue as long as the respective action is active. Make an entry in the configuration file to replace any of these effects with a new one. Or to turn off the effect add an entry that references the fx_dummy effect (which does nothing).

wake The wake effect name. No fx_wake False Airbus A321( wake=fx_wake )
water The landing, taxiing or taking off from water effect. No fx_spray False Airbus A321( water=fx_spray )
waterspeed Traveling at speed on the water. No fx_spray False

touchdown The touchdown effect, which usually is followed by an optional 1 to indicate the effect is to be run once only. No fx_tchdwn True Airbus A321( touchdown=fx_tchdwn, 1 ) Aircreation582SL( touchdown=fx_tchdwn_s, 1 )

As with [Lights], effects can be specified for an aircraft within a list like this:

effect.n = \<nodename\>#\<SimCode\>#<fxFileName>`

Where n is the effect number and nodename is the model node that the effect will be attached to. SimCode is a [script expression][script-postfix] that has to return true or false (1 or 0), the effect will show up based on this value.

For example:

effect.0 = L1_EngineFX#(A:GENERAL ENG COMBUSTION:1, boolean)#fx_reactorHeat
      effect.1 = L2_EngineFX#(A:GENERAL ENG COMBUSTION:2, boolean)#fx_reactorHeat
      effect.2 = R1_EngineFX#(A:GENERAL ENG COMBUSTION:3, boolean)#fx_reactorHeat
      effect.3 = R2_EngineFX#(A:GENERAL ENG COMBUSTION:4, boolean)#fx_reactorHeat

The last input in the line is a .fx file. At this time, it is not yet possible to re-define the effects used.

Full example for a Boeing:

[EFFECTS]
      wake=fx_wake
      water=fx_spray
      dirt=fx_tchdrt
      concrete=fx_sparks
      touchdown=fx_tchdwn, 1
      effect.0 = L1_EngineFX#(A:GENERAL ENG COMBUSTION:1, boolean)#fx_reactorHeat
      effect.1 = L2_EngineFX#(A:GENERAL ENG COMBUSTION:2, boolean)#fx_reactorHeat
      effect.2 = R1_EngineFX#(A:GENERAL ENG COMBUSTION:3, boolean)#fx_reactorHeat
      effect.3 = R2_EngineFX#(A:GENERAL ENG COMBUSTION:4, boolean)#fx_reactorHeat

You can find more information about fx files here.

[LOADING]

[FLTSIM]

Configuring a specific variation for parked planes, air traffic and multiplayer

In the FLTSIM section each variation of a plane can specify its intended uses.

Property Description Possible Values Default Examples
isAirTraffic If it is true (1) this variation is considered to be optimized for use as a parked plane, or as part of air traffic. 0, 1 0 “Airbus A320 Neo Asobo AirTraffic 00” variation (“isAirTraffic = 1”)
isUserSelectable If it is false (0) this variation is not selectable by the user to fly. 0, 1 1 “Airbus A320 Neo Asobo AirTraffic 00” (“isUserSelectable = 0”)
icao_airline Non-localised ICAO code of the airline if the variation is a livery for a real airline. All real airline ICAO codes. Empty Generic Airliner Twin Engines (icao_airline = “UAL”)

Setting the “isAirTraffic” flag to true (1) allows the game to use that variation as a parked (grounded) passive aircraft, and potentially as a part of the “Real Online” air traffic. If a plane variation is marked as isAirTraffic, it should be an optimized variation, with a lower level of detail, and therefore it should also be marked as “isUserSelectable=0”. Aircrafts that have the icao_generic flag set to True (1) should have all variations flagged as not selectable by the user (“isUserSelectable=0”) and part of the air traffic (“isAirTraffic=1”).

Variations marked with the “isAirTraffic” flag will appear in the game as parked aircrafts (depending on parking spot compatibilities) if the user sets the traffic type to “Real Online” and/or they turn the “Use Only Generic/Optimized Plane Models for Traffic” option ON in the graphic settings.

Variations with a non-empty “icao_airline” field are considered real airline liveries. They will be used for parked planes, but for Real Online planes they will only match with flight identifications that carry the airline ICAO code of that variation. For example, if a flight has identification “UAL1337” the system will first look for variations of the matching plane that have icao_airline = “UAL”, and if none are found, it will use any variation with an empty icao_airline.

If the GENERAL section of the Aircraft Configuration File contains contains correct ICAO information, it will be usable as part of the live air traffic, to represent Real Online planes that match their Aircraft Type Designator. Even if a plane does not have any air traffic (“isAirTraffic=1”) variations, there is still a chance it might be used to represent a Real Online plane if the user set the “Use Only Generic/Optimized Plane Models for Traffic” option to OFF in the graphic settings. The variations of an aircraft that are marked with the isAirTraffic flag will always be prioritized over the other variations (even airline liveries) both to use as parked planes and to swap Real Online planes.

Please check the AI configuration file documentation for additional informations on setting up a passive aircraft. In addition to that, even aircrafts flagged as “generic” or with only air traffic variations need correct values for most of the fields in the Flight Model configuration file. Please verify, among others, information on contact points, static center of gravity height, static pitch, reference speeds.
