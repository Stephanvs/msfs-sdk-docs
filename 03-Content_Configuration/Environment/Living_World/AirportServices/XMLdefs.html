Here are defined the specifications of the Services files, by first considering the main file Services.xml, and then specificities of Service and IdleWorker script files

Services.xml

<SimBase.Document>

Base of an xml document.

Element Description Quantity
<Service.ParkingSpaceServices> List of the parking space services 0-1
<Service.AirportServices> List of the global airport services 0-1

<Service.ParkingSpaceServices>

List of the parking space services

Element Description Quantity
<ParkingSpace> An entry of parking space definition, with one or several parking types. 1+

Note that each parking type must appear in at most one of the ParkingSpace in the list.

<ParkingSpace>

A description of a general category of parking space, including one or several types.

Element Description Quantity
<ParkingTypes> List of parking types corresponding to this entry 1
<IdleWorkers> List of IdleWorkers for this entry 0-1
<Services> List of Services for this entry 0-1
<SleepingActivities> Parameters for ParkedAircrafts activities 0-1
Attribute Description Type Required
name Name of this ParkingSpace entry String No
MinParkingSpaces Minimum number of parkings (of any type) on the airport to take this into account. Useful to avoid having services on micro-airports with only 1 parking. Integer No

<ParkingTypes>

Lists the types of parking corresponding to this ParkingSpace.

Element Description Quantity
<ParkingEntry> A single parking type 1+

<ParkingEntry>

A single parking type.

Attribute Description Type Required
ParkingType Enum of a parking type (see TaxiwayParkings Enum Yes

<IdleWorkers>

List of IdleWorkers for this ParkingSpace.

Element Description Quantity
<Spot> A position to spawn one IdleWorker script among several available 0+

<Spot>

Element Description Quantity
<IdleWorker> A IdleWorker script that can be placed in this Spot 1+
Attribute Description Type Required
Frequency Frequency (0.0-1.0) at which an IdleWorker will be created at this Spot Float Yes
SpawnDistanceParkingRadiusFactor Determines the spawning position of the Spot by multiplying this factor by the radius of the parking Float Yes
SpawnDistanceAdditionalOffset Determines the spawning position of the Spot by adding this offset in meters (can be negative to be inside the parking) Float Yes
SpawnAngleFromParking Determines the spawning position of the Spot by determining at which angle from the parking to move the specified distance. Float Yes
HeadingFromParkingHeading Determines the heading of the spawned IdleWorker, relative to parking heading (and not North) Float Yes

Each parking in each airport is represented by a circle (with center and radius), a type, and a heading (orientation relative to North) . From this, the spawning position of an IdleWorker Spot is determined by three parameters. First, this position will be at a distance from the center of the parking equal to SpawnDistanceParkingRadiusFactor * (radius of current parking) + SpawnDistanceAdditionalOffset. Then, the position will be computed from the center of parking and distance by moving along a line determined by an angle SpawnAngleFromParking from the parking heading. See the examples chapter for more clarity.

<IdleWorker>

Defines one of the several IdleWorker scripts that can be used for this Spot, along with its relative weight.

Attribute Description Type Required
Filename Name of the script file. Should be the full path from the Airport_Services root folder (not included in the path), and not include the file extension. String Yes
EntryWeight Relative weight determining how likely it is to be chosen over other scripts for this Spot (higher weight means more likely to be chosen) Integer Yes

Parking <Services>

List of Services for this ParkingSpace

Element Description Quantity
<Service> A service along with its spawning parameters 0+

Parking <Service>

A single Service to create at this parking.

Attribute Description Type Required
name Informal name of the Service String Yes
Filename Name of the script file. Should be the full path from the Airport_Services root folder (not included in the path), and not include the file extension. String Yes
AIType Enum corresponding to the AI Type of the main object in this Service. Enum Yes
Frequency Frequency (0.0-1.0) at which this Service will be created Float Yes
SpawnDistanceParkingRadiusFactor Determines the spawning position of the Service by multiplying this factor by the radius of the parking Float Yes
SpawnDistanceAdditionalOffset Determines the spawning position of the Service by adding this offset in meters (can be negative to be inside the parking) Float Yes
SpawnAngleFromParking Determines the spawning position of the Service by determining at which angle from the parking to move the specified distance. Float Yes
HeadingFromParkingHeading Determines the heading of the spawned Service object, relative to parking heading (and not North) Float Yes

Each parking in each airport is represented by a circle (with center and radius), a type, and a heading (orientation relative to North) . From this, the spawning position of a Service is determined by three parameters. First, this position will be at a distance from the center of the parking equal to SpawnDistanceParkingRadiusFactor * (radius of current parking) + SpawnDistanceAdditionalOffset. Then, the position will be computed from the center of parking and distance by moving along a line determined by an angle SpawnAngleFromParking from the parking heading. See the examples chapter for more clarity.

<SleepingActivities>

Determines parameters for ParkedAircrafts and former AirTraffic aircrafts that are no longer part of the AirTraffic.

Element Description Quantity
<SleepingAirlinerServices> Parameters regarding how often parked airliners should randomly decide to use Services. 0-1
<SleepingSmallPlaneRefueling> Parameters regarding how often parked GA aircrafts should randomly decide to taxi to a fueling station before going back to its parking. 0-1

<SleepingAirlinerServices>

Attribute Description Type Required
Timer How often (in seconds) will a given airliner randomly decide to use Services. Float Yes
Probability Probability (0.0-1.0) the airliner will decide to use Services when its Timer expires. Float Yes

Note that the Timer is always restarting after expiring, which means that byt waiting long enough, each airliner will use Services multiple times. We recommend using a Timer of at least 300 seconds and a Probability of less than 0.4, to avoid having too many aircrafts using Services all the time.

<SleepingSmallPlaneRefueling>

Attribute Description Type Required
Timer How often (in seconds) will a given GA aircraft randomly decide to go refueling. Float Yes
Probability Probability (0.0-1.0) the GA aircraft will decide to go refueling when its Timer expires. Float Yes

Note that the Timer is always restarting after expiring, which means that byt waiting long enough, each GA aircraft will go refueling multiple times. We recommend using a Timer of at least 300 seconds and a Probability of less than 0.2, to avoid having too many aircrafts using the few Fuel stations all the time.

<Service.AirportServices>

List of the global airport services

Element Description Quantity
<Services> List of Services 1

Airport <Services>

List of Airport Services

Element Description Quantity
<Service> A service along with its spawning parameters 1+

Airport <Service>

A single Airport Service to create.

Attribute Description Type Required
name Informal name of the Service String Yes
Filename Name of the script file. Should be the full path from the Airport_Services root folder (not included in the path), and not include the file extension. String Yes
ParkingType Which type of parking will this Service be created in. Enum Yes
Frequency Frequency (0.0-1.0) at which this Service will be created Float Yes

Service and IdleWorker script files

Both Service and IdleWorker script files are some form of Mission xml files. The root of the xml file should be a <SimBase.Document>

Script <SimBase.Document>

Attribute Description Type Required
Type Must contain MissionFile String Yes
Element Description Quantity
<WorldBase.Flight> The main part of the script file 1

<WorldBase.Flight>

The main part of the script file. The element list here is incomplete, and any element from missions is available. The following list just highlights the specificities for Service and IdleWorker scripts.

Element Description Quantity
<Service.GroundService> Specific to Services, list of service states with associated Actions, triggering the Actions when the service enter the corresponding state. 0-1
<SimMission.SendMessageToAIAction> Only used by Services, an Action that sends a message to the AI code in MSFS of an object. 0+
<SimContain.Container> Create a SimObject, most notably to create the main object of the Service/IdleWorker, and possibly the Human pilot of the vehicle. 1+
Any other element from a mission, see corresponding chapter

<Service.GroundService>

Element Description Quantity
<States> List of service states with associated Actions, triggering the Actions when the service enter the corresponding state. 1

<States>

Element Description Quantity
<State> A single state for a Service object, with its list of Actions 1+

<State>

Element Description Quantity
<ObjectReferenceList> List of references to Actions to be executed when the service enter this state. 1
Attribute Description Type Required
name Name of the state, see below for the list of available states String/Enum Yes
UniqueRefId Reference the GUID of the corresponding service object (InstanceId attribute of the corresponding <SimContain.Container>) GUID Yes

Here is the list of states available for each AIType of object. Objects that have an AIType not in the list have no state triggers.

State name Description Available AIType
Sleep State at rest, when the vehicle finished its task and turns off its engine. Can also be triggered if some error occured, and the vehicle just stops its normal behavior. BaggageLoader, BaggageCart, BoardingRamp, CateringTruck, GroundPowerUnit, Pushback, SmallPushback, FuelTruck
Move_Toward_Aircraft The vehicle has been requested and starts moving towards the aircraft BaggageLoader, BaggageCart, BoardingRamp, CateringTruck, GroundPowerUnit, Pushback, SmallPushback
Before_Reaching_Aircraft Shortly before reaching the aircraft, the vehicle has some action to perform (BaggageLoader inclines its ramp, BoardingRamp extends its stairs, CateringTruck elevates its container) BaggageLoader, BoardingRamp, CateringTruck
Reached_Aircraft Final state upon reaching aircraft (BaggageLoader and BoardingRamp finish moving, CateringTruck extends its bridge, GroundPowerUnit deploys its cable) BaggageLoader, BoardingRamp, CateringTruck, GroundPowerUnit, FuelTruck
Loading The vehicle is connected to the aircraft and starts doing its job BaggageLoader, BaggageCart, BoardingRamp, CateringTruck, GroundPowerUnit, Pushback, SmallPushback
Unloading (future) Same as Loading, but aimed to take place after a flight (to unload baggages and passengers) rather that before a flight Not yet used, should be added in the future
Finished_Operation Immediately after the Loading/Unloading state ended, the vehicle starts moving back or animating to detach from the aircraft BaggageLoader, BoardingRamp, CateringTruck, GroundPowerUnit, Pushback, SmallPushback, FuelTruck
Before_Leaving_Aircraft Final gathering up state before the vehicle completely leave the aircraft (BaggageLoader moves back, BoardingRamp retracts its stairs, CateringTruck moves down its container) BaggageLoader, BoardingRamp, CateringTruck
Move_Back_To_Parking The vehicle moves back to its initial position BaggageLoader, BaggageCart, BoardingRamp, CateringTruck, GroundPowerUnit, Pushback, SmallPushback
Loading_AI Equivalent to Loading, when connected to an AI aircraft FuelTruck
Loading_User Equivalent to Loading, when connected to the User aircraft FuelTruck
UserClosedWindowDuringLoading When the FuelTruck starts Loading_User, the refueling window appears on screen. This state is triggered if the User closes this window during the Loading state. FuelTruck

Finally, the FuelTruck also has the following states available, but with different meaning: Move_Toward_Aircraft, Move_Back_To_Parking, Before_Reaching_Aircraft, Before_Leaving_Aircraft. The first two are used when the FuelTruck starts taxiing accross the airport towards an aircraft (it can perform multiple requests in a row) or its original parking. The last two are used to move from the parking edge to the aircraft, and back.

State <ObjectReferenceList>

List of the Actions to be executed when the service enter this state.

Element Description Quantity
<ObjectReference> A single reference to an Action. 0+

State <ObjectReference>

A reference to a single Action to be executed when the service enter this state.

Attribute Description Type Required
id Informal description of the action. Not an actual identifier, but make scripts much more readable. String Yes
InstanceId Reference the GUID of the corresponding Action GUID Yes

See the mission chapter for more details on Actions.

<SimMission.SendMessageToAIAction>

When executed, this action will send a message to the AI of an object (or multiple objects).

Element Description Quantity
<ObjectReferenceList> List of references to objects to which send the message 1
Attribute Description Type Required
Name Message to be sent String/Enum Yes

A message sent to an AI could be any string, but only a few messages have any effect, and not on all AIType. Here is a list of the current valid messages, with the AI types that interpret them:

Message Description Available AIType
FINISHED_OPERATION Warns the AI that the Loading (or Unloading) phase is finished, and that this service should start its regular behavior to detach from aircraft. BaggageLoader, BoardingRamp, CateringTruck, GroundPowerUnit, FuelTruck

Currently, no other message has any effect. This list will be expanded when new messages with effect are added.

SendMessageToAIAction <ObjectReferenceList>

List of the objects with AIs to which the message will be sent.

Element Description Quantity
<ObjectReference> A single reference to an object. 1+

SendMessageToAIAction <ObjectReference>

A reference to a single object with AI to which the message will be sent.

Attribute Description Type Required
id Informal description of the object. Not an actual identifier, but make scripts much more readable. String Yes
InstanceId Reference the GUID of the corresponding object GUID Yes

<SimContain.Container>

Create a SimObject. More details are available in the mission chapter, but here are a few especially useful attributes and elements for Services and IdleWorkers.

Attribute Description Type Required

Descr Informal description of the SimObject String Yes
AIType AI type of the SimObject. Enum Yes
UseRegionContainerType If true and if no ContainerTitle is specified, the model used will be taken at random in the correct category (based on AIType) from the current LivingWorld region Boolean No
ContainerTitle Explicit title for the SimObject. Only required if UseRegionContainerType is false or undefined. String No
Any other attribute as defined in the mission chapter

The <SimContain.Container> can also have the following elements:

Element Description Quantity
<HumanPilotAI> For SimObject with an AIType equal to GroundVehicle_Pilot or Aircraft_Pilot, defines the specific properties of this AI. 0-1
<IdleWorkerAI> For SimObject with an AIType equal to IdleWorker, defines the specific properties of this AI. 0-1
<AttachedWorldObject> Actually attach this SimObject to another, or just initialize it to a relative position. 0-1
Any other element as defined in the mission chapter

<HumanPilotAI>

AI parameters for objects with AIType GroundVehicle_Pilot or Aircraft_Pilot.

Attribute Description Type Required
HumanPilotType Determines which type of pilot is this object, and thus which parameters should be read from the cfg file of the attached vehicle (attach node and default animation). Enum can be Unknown, Pilot, or Copilot. Enum No

Note that such a pilot object must have an <AttachedWorldObject> in order to be actually attached to its vehicle.

<IdleWorkerAI>

AI parameters for objects with AIType IdleWorker.

Attribute Description Type Required Default
Unit_Mode Initial Unit Mode of the AI. Should not make any difference for IdleWorker with current version of MSFS. Enum No Waypoint
GroundCruiseSpeed Default speed in knots for this IdleWorker AI Float No 35
GroundTurnSpeed IdleWorker AI slows to this speed in knots when making sharp turns Float No 10
GroundTurnTime IdleWorker AI attempts to make 90 degree turns in this amount of time (seconds) Float No 3.5
WaypointTouchDistance Distance in meters at which IdleWorker AI considers waypoint to be reached Float No 1.0
YieldToUser Whether the IdleWorker will stop moving when getting too close to the User aircraft Boolean No True
KeepLastHeading Whether the IdleWorker will try to reach the heading defined by its last two waypoints before reaching its last waypoint, even if that means slightly less precision on the final position. When false, the IdleWorker will only focus on reaching the exact position of the last waypoint, ignoring orientation. Boolean No True
CanReverse Whether the IdleWorker will try to maneuver backwards when it missed one if its waypoints. Boolean No True

Note that the CanReverse attribute is better set to False for IdleWorker, as it is more useful for vehicles.

Element Description Quantity
<WaypointList> List of waypoints the IdleWorker with start with. Will only be useful if the Unit_Mode is set to Waypoint. 0-1

Unless using simple IdleWorker scripts that do nothing except for creating an IdleWorker, we recommend not using the WaypointList in the <IdleWorkerAI>. It is usually better to initialize the IdleWorker as idle, and use scripting (such as TimerTrigger for example) to later set up various behaviors, including a WaypointList by using a ActivateWaypointsAction.

<AttachedWorldObject>

This element can have two different behaviors, which can be confusing. We might change this to clarify this confusion in the future.

Element Description Quantity
<ObjectReference> Reference to the object this is attached to. 1
Attribute Description Type Required
OffsetXYZ Offset XYZ (in meters) defining where to place this object relative it its reference. 4 comma-separated floats No
Orientation PBH orientation (in degrees) of this object relative to its reference. 3 comma-separated floats No
IsOnGround Is this object placed on ground Boolean No
NodeName Name of a Node from the reference object to which this object will be attached. String No

Note OffsetXYZ 4th value is unused, and axes X, Y, Z represent the left-right axis, upward axis, and forward axis, respectively. For Orientation, currently only the heading (3rd value) is taken into account, but this might change in the future.

If the object has a NodeName defined here, or if it has a <HumanPilotAI> with a HumanPilotType Pilot or Copilot, then it will be actually attached to the reference (to a specific Node, either explicit, or defined in the cfg of the reference model). Otherwise, only the initial position will be used to initialize this object relative to the reference, using OffsetXYZ, Orientation, and IsOnGround informations.

AttachedWorldObject <ObjectReference>

Reference to a main object to which attach another object, or only initialize this other object in a relative position.

Attribute Description Type Required
id Informal description of the object. Not an actual identifier, but make scripts much more readable. String Yes
InstanceId Reference the GUID of the corresponding Object GUID Yes
